# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Identity

G1_Hand is an embedded firmware project for a **robotic dexterous hand controller** running on the **HPMicro HPM6E80** RISC-V MCU (`rv32imac` with `Zba_Zbb_Zbc_Zbs` extensions). It targets the `hpm6e00_ethercat_slave` board and builds against **HPM SDK v1.11.0** (localized in-repo).

## Build System

- **Build tool**: CMake (≥3.13) with Ninja generator
- **Toolchain**: `riscv32-unknown-elf-gcc` 13.2.0 (requires `GNURISCV_TOOLCHAIN_PATH` in the SDK environment)
- **SDK**: HPM SDK v1.11.0 is localized at `hpm_sdk_localized_for_hpm6e00_ethercat_slave/` — `HPM_SDK_BASE` is hardcoded by CMakeLists.txt line 10, do **not** set it externally
- **Board**: `hpm6e00_ethercat_slave` (forced by CMakeLists.txt line 17 — reconfiguring with any other `-DBOARD=` will fail)

### Building

The build directory is pre-configured at the repo root:

```sh
cd hpm6e00_ethercat_slave_flash_xip_debug
ninja
```

To reconfigure from scratch:

```sh
cd hpm6e00_ethercat_slave_flash_xip_debug
cmake .. -G Ninja
ninja
```

### Key build details

- Linker script: `flash_xip.ld` (execute-in-place from flash: 16 MB flash at 0x80000000, 32 MB SDRAM)
- Stack/heap: 16 KB each
- C standard flags: `-g -Wall -Wundef -Wno-format -fomit-frame-pointer -fno-builtin -ffunction-sections -fdata-sections`
- Architecture flags: `-march=rv32imac_zicsr_zifencei -mabi=ilp32`
- Linker flags: `-static -nostartfiles -Wl,--gc-sections --specs=nano.specs -u _printf_float -u _scanf_float`
- IDE projects for IAR EW and Segger Embedded Studio are auto-generated by `generate_ide_projects()` in CMakeLists.txt

## Adding New Files

**Critical**: Every new `.c` file must be explicitly registered in `CMakeLists.txt` via `sdk_app_src()`. Every new include path must be added to `target_include_directories(app PRIVATE ...)`. The `generate_ide_projects()` function auto-discovers files **only** for IDE project files (Segger/IAR) — it does **not** affect Ninja builds.

## Architecture

The project follows a layered embedded firmware structure with **no RTOS** — all tasks are cooperative poll-loop driven from `main()`'s `while(1)`.

```
user_app/                          — Application layer
  app_main.c                       — main() entry, calls board_init() then poll-loop
  tasks/                           — Cooperative poll tasks
    led_task.c                     — LED status indicator (wires drv_led → service/led)
    uart_task.c                    — RS-485 comms task (currently disabled in main)
    finger_task.c                  — Finger motor multi-port RS-485 scheduler
    canfd_task.c                   — CAN-FD send/receive test task (CAN4, PA16/PA17/PA18)
drivers/
  bsp/                             — Board support package (thin HAL wrappers)
    bsp_gpio.c                     — GPIO init/read/write/toggle (wraps hpm_gpio_drv)
    bsp_uart.c                     — UART HAL: DMA send/recv, blocking I/O, HDMA ISR, idle detection
  device_driver/                   — Peripheral device drivers
    drv_led.c                      — PF02 GPIO LED control (on/off/toggle/set/read)
    drv_rs485.c                    — 3-port RS-485 half-duplex driver (interrupt-driven TX queue + DMA ring RX)
    drv_can.c                      — Multi-instance CAN-FD driver (MCAN ISR + kfifo RX, blocking/non-blocking TX)
middlewares/
  utils/                           — Data structure libraries
    kfifo.c                        — Linux-kernel-style ring buffer (power-of-2, thread-safe via MSTATUS CSR)
    clist.c                        — Intrusive doubly-linked circular list (Linux list.h style)
  fsm/                             — Generic finite state machine framework
    fsm.c                          — O(1) transition matrix, entry/exit callbacks, guard conditions
  protocol_tools/                  — Protocol framing utilities
    protocol_packer.c              — Frame builder: [header][payload][checksum][footer]
    protocol_parser.c              — Frame parser with idle timeout, header/footer match, kfifo input
service/                           — Service modules (higher-level logic on top of drivers)
  led.c                            — FSM-based LED controller: ON/OFF/BLINK_CODE, async command queue, hot-updatable params
board/hpm6e00_ethercat_slave/      — Board-level init and pinmux (HPM SDK)
```

### Dependency graph

```

  ├── led_task

  │     └── service/led

  ├── finger_task

  │     ├── protocol_parser ── kfifo

  └── uart_task (disabled)

```

### Key architectural patterns

1. **Cooperative polling**: `main()` calls `task_init()` once, then loops `task_poll()` for each active task. No preemptive scheduler — each poll call must return quickly.
2. **RS-485 TX is fully interrupt-driven**: Data enqueued to kfifo → DMA kicked. HDMA ISR chains subsequent chunks. When queue empties, UART is flushed and DE dropped. Zero application-level TX polling.
3. **RS-485 RX uses double buffering**: DMA writes to 128-byte circular buffer → hardware idle-detect triggers ISR → data copied to 256-byte kfifo → application reads from kfifo.
4. **FSM-based services**: `service/led` uses `middlewares/fsm` for state management with async command queues (kfifo). Each LED instance has its own FSM context. This pattern is intended for future service modules.
5. **Zero dynamic allocation at runtime**: All memory (FSM tables, kfifo buffers, DMA descriptors) is statically allocated. `__malloc` is used sparingly and always followed by `memset(ptr, 0, size)`.

## Coding Conventions

**必须严格遵循 [MODULE_CODING_GUIDE.md](MODULE_CODING_GUIDE.md) 中的全部规范生成代码。** 以下为关键规则摘要，详细模板和示例请参考原文档。

### 代码风格

- **缩进**: 4 空格，不使用 Tab
- **大括号**: 函数定义左大括号另起一行 (Allman)；控制语句 (if/for/while) 左大括号与关键字同行 (K&R)
- **行长**: 单行不超过 100 字符
- **标准**: 遵循 WebKit 代码风格 + MISRA C:2012

### 类型与命名

- **整数类型**: 必须使用 `<stdint.h>` 固定宽度类型 (`uint8_t`, `uint16_t`, `uint32_t`, `int32_t`)
- **布尔类型**: 必须使用 `<stdbool.h>` 的 `bool` (值 `true`/`false`)
- **const 正确性**: 输入只读参数必须加 `const`；返回只读指针必须加 `const`

| 类型 | 命名规范 | 示例 |
|------|----------|------|
| 枚举类型 | `模块名_xxx_t` | `protocol_parser_error_t` |
| 结构体类型 | `模块名_xxx_t` | `protocol_parser_config_t` |
| 函数指针类型 | `模块名_xxx_cb_t` | `protocol_parser_check_cb_t` |
| 公共函数 | `模块名_动词` | `protocol_parser_init()` |
| 私有静态函数 | `模块名_功能描述` | `parser_check_header()` |
| 局部变量 | 小写蛇形 | `payload_len` |
| 全局/静态变量 | 小写蛇形 (`s_` 前缀) | `s_uart1_rx_buffer` |
| 宏/常量 | 大写蛇形 | `IDLE_TIMER_US` |

**源文件模板 (.c)** — 代码组织必须严格按以下顺序:

```c
// 1. 文件头注释 (@file, @author, @version, @brief, @attention + BSD-3-Clause)
// 2. /* Includes ------------------------------------------------------------------*/
// 3. /* Private constants ---------------------------------------------------------*/

// 5. /* Private function prototypes -----------------------------------------------*/
// 6. /* Exported functions --------------------------------------------------------*/
// 7. /* Private functions ---------------------------------------------------------*/
```

### 模块设计模式 (context/config)

每个模块必须使用双结构体模式：

**配置结构体** — 包含初始化参数和回调函数（init 时确定，之后不变）：

```c
typedef struct {
    const char *name;                    /**< 模块名称 */
    const uint8_t *header;               /**< 帧头数据 */
    uint16_t header_len;                 /**< 帧头长度 */
    module_check_cb_t check_cb;          /**< 校验回调（属于配置） */
} module_config_t;
```

**上下文结构体** — 运行时状态，嵌套配置实例：

```c
typedef struct module_context module_context_t;
struct module_context {
    module_config_t config;              /**< 配置参数（嵌套） */
    void *fifo;                          /**< 内部FIFO缓冲区 */
    bool initialized;                    /**< 初始化标志 */
};
```

访问方式：`ctx->config.xxx` (配置), `ctx->xxx` (运行时状态)。

### 注释规范

- **所有公共 API 和复杂逻辑必须使用中文注释**
- 函数: Doxygen 风格 — `@brief`, `@param`, `@return`
- 枚举值: 行尾 `/**< 中文说明 */`
- 结构体成员: 行尾 `/**< 中文说明 */`
- 行内注释: `// 中文说明`
- 枚举第一个值必须为 `MODULE_NAME_OK = 0`

### 错误码定义

```c
typedef enum {
    MODULE_NAME_OK = 0,                  /**< 操作成功 */
    MODULE_NAME_ERROR_NULL_PTR,          /**< 空指针错误 */
    MODULE_NAME_ERROR_INVALID_PARAM,     /**< 无效参数 */
    MODULE_NAME_ERROR_UNINITIALIZED,     /**< 未初始化 */
} module_name_error_t;
```

### 函数设计规范

**初始化函数** — 必须按以下步骤实现：

1. 检查参数有效性 (null ptr)
2. 处理特殊情况 (如 NULL header → 长度置 0)
3. 检查配置有效性
4. 如果已初始化，先调用 deinit
5. 整体复制配置 (`ctx->config = *config`)
6. 初始化运行时状态
7. 分配资源
8. 标记 `ctx->initialized = true`

**参数检查模板**:

```c
if (!ctx || !config) { return MODULE_ERROR_NULL_PTR; }
if (!ctx->initialized) { return MODULE_ERROR_UNINITIALIZED; }
if (len == 0) { return MODULE_ERROR_INVALID_PARAM; }
```

**未使用参数必须显式标记**: `(void)param;`

### 内存管理

**动态分配内存必须立即清零** —— 未初始化内存可能导致死机：

```c
// ✅ 正确
module_context_t *ctx = (module_context_t *)__malloc(sizeof(module_context_t));
if (ctx == NULL) { return NULL; }
memset(ctx, 0, sizeof(module_context_t));  // 必须紧跟

// ❌ 错误 — ctx->initialized 可能是垃圾值，导致 deinit 访问垃圾指针死机
module_context_t *ctx = (module_context_t *)__malloc(sizeof(module_context_t));
module_init(ctx, &config);
```
